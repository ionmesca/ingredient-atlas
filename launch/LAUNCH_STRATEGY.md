# Launch Strategy

## Positioning

Ingredient Atlas is open ingredient images and metadata for people building recipe, grocery, pantry, and meal-planning apps.

The simple line:

> I made this because every recipe app needs ingredient images, but sourcing clean, consistent, reusable ones is annoying.

## Who It Is For

- recipe app builders
- grocery app builders
- pantry tracker builders
- meal-planning tools
- Cooklang ecosystem projects
- UI prototypers
- dataset people who care about food data

## What Is Different

- Food-101: dish classification, not ingredient UI assets.
- Open Food Facts: packaged products and labels, not clean isolated ingredients.
- Recipe1M: recipe and dish research, not app-ready ingredient thumbnails.
- Roboflow-style datasets: usually detection or model training first.
- Ingredient Atlas: stable ingredient slugs, clean images, aliases, categories, checksums, hosted files, and correction paths.

## What Usually Fails

- posting once and waiting;
- sounding like an ad;
- hiding the limitations;
- shipping without a demo;
- asking for stars;
- posting the same thing in every community.

## What Actually Works

- show the real images;
- show copy-paste usage;
- be honest that images are AI-generated;
- invite corrections;
- send thoughtful notes to relevant builders;
- reply fast when people find issues.

## Launch Sequence

1. Make Hugging Face public.
2. Make GitHub public.
3. Publish npm package.
4. Create GitHub release.
5. Post Show HN.
6. Post r/datasets.
7. Post a Hugging Face community note.
8. Send 10 direct outreach notes.
9. Wait 24 to 48 hours, reply, fix obvious issues.
10. Send the next 10 notes.

## Measure

- Hugging Face likes, downloads, and discussions;
- GitHub stars, issues, forks;
- npm downloads;
- inbound corrections;
- people actually using image URLs in projects.

The best signal is not stars. The best signal is someone saying, “I used this in my app.”
